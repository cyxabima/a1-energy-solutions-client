# A1 Energy Solutions — Frontend

## Project Overview

A1 Energy Solutions — mini ERP system. This repository contains the **client-side only** frontend code.

### Phase 1 — Inventory System (Current)
- Inventory management with **ownership incorporation** — multiple owners per product with associated amounts
- Metadata-driven product system where **categories define product attributes**
- Admin creates categories and defines attribute schemas (e.g. solar inverter → brand, power rating, amper type, IP rating)
- Admin sets allowed values for each attribute — one-time effort, then users select from defined values only
- **Brands** and **Units** are standalone entities (general attributes, not per-category)
- Products belong to a category; product name is auto-generated by concatenating all attribute values with spaces
- Only buying price is stored on the product; sale price is entered at the counter (bargaining is available)
- Confirming: products are currently considered under categories (ownership rules may evolve)

### Phase 2 — Sales & HR (Planned)
- Full customer/client management
- Salesman management
- Retail sales
- Full invoicing and billing
- To be discussed and designed after Phase 1 implementation

## Tech Stack

| Layer | Technology |
|-------|-----------|
| Framework | React 19 + Vite 8 |
| Language | TypeScript 6 |
| Routing | TanStack Router (file-based routing, **no SSR**, no TanStack Start — just Vite + TanStack Router) |
| UI Components | shadcn/ui (with `@base-ui/react` primitives) |
| Styling | Tailwind CSS v4 — **never use CSS files, Tailwind only** |

| Compiler | React Compiler enabled (via `babel-plugin-react-compiler`) |
| Package Manager | pnpm |

## Architecture

- **Client-side only** — no SSR, no server actions, no Next.js patterns
- **All data operations** go through API calls to the backend at `http://localhost:8000`
- **Authentication** — JWT via HTTP-only cookies
- **No CSS files** — every style must be done with Tailwind utility classes
- Path alias `@/` resolves to `src/`
- Auto code-splitting enabled via TanStack Router plugin

## Domain Model

### Categories
- Metadata-driven entity — the fundamental organizing principle
- Admin creates categories and defines what attributes products in that category must have
- Example: "Solar Inverter" category → attributes: brand, power rating, amper type, IP rating, etc.

### Attributes
- Defined per category by admin
- Admin specifies allowed values for each attribute (e.g. power ratings: 1kW, 2kW, 5kW, etc.)
- Editable — new types/values can be added later
- One-time setup effort, after which users only select from predefined values (no garbage data)

### Brands (Standalone Entity)
- General attribute — not tied to any specific category
- Separate entity for reuse across all categories

### Units (Standalone Entity)
- General attribute — not tied to any specific category
- Separate entity for reuse across all categories

### Products
- Must belong to a category
- Name is auto-generated: concatenation of all attribute values with spaces
- Stores **buying price** by default
- Sale price is entered at the counter (supports bargaining)
- Ownership model: multiple owners with associated amounts (to be confirmed)

## Conventions

- Use file-based routing with TanStack Router under `src/routes/`
- Components go in `src/components/` (UI components in `src/components/ui/`)
- Store files in `src/store/` using Zustand
- API/service layer in `src/lib/` or `src/services/`
- All new UI components added via `shadcn` CLI
- Follow React Compiler best practices (no unnecessary useMemo/useCallback)
- Strict TypeScript — `noUnusedLocals`, `noUnusedParameters` enabled
